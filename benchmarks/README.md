# benchmarks

Overworld 框架的性能基准与回归守护。所有基准都直接跑在各包**构建产物**（`dist`）上，
覆盖:事件总线、注册表、A* / HPA* 寻路、Agent 更新循环、行为树、任务引擎、
碰撞解算、网络快照缓冲与客户端预测。

## 运行方式

```bash
# 先确保各包已构建(仓库根目录)
pnpm build

# 运行全部基准:打印对齐表格,并写入 results/latest.json
cd benchmarks
pnpm bench

# 运行回归守护(vitest,计数型断言)
pnpm test
```

表格列说明:`ops/sec`(每秒操作数,取多轮中位数)、`mean`(单次操作平均耗时)、
`Δ ops/s`(相对 `baseline.json` 的百分比变化,存在基线时才显示)、`meta`(工作负载参数)。

所有工作负载使用种子化 PRNG(mulberry32),因此每次运行的输入完全一致,
数字波动只来自机器本身。

## 基线更新

```bash
pnpm bench -- --update-baseline   # 把 results/latest.json 复制为 baseline.json
```

`baseline.json` 建议提交进仓库:之后每次 `pnpm bench` 都会自动打印 Δ% 对比列。
在有意的性能优化 / 已接受的性能取舍落地后更新基线;不要在噪声波动后更新。

## 为什么守护是计数型而非时间型

`src/__tests__/guards.test.ts` 里的回归守护**故意不断言耗时**:

- 时间受机器、CI 负载、JIT 分层、GC 时机影响,同一份代码在不同环境下差数倍,
  基于时间的阈值不是漏报就是误报(flaky)。
- 计数不变量只依赖算法本身,跨机器 100% 确定:算法退化时必然被抓住,
  环境再慢也不会误报。

当前守护:

1. 200×200 障碍地图上,HPA* 展开的格子数 < 普通 A* 的一半(复用 `stats` API);
2. 空旷场地上,寻路路径长度 ≤ 直线下界的 1.3 倍(A* 与 HPA* 都查);
3. 事件总线:0 监听器时零次调用;N 个监听器每次 emit 恰好各被调用一次;
4. 任务引擎:1000 个匹配事件恰好产生 1000 点进度(不重复计数、不丢失),
   filter 不匹配的任务进度为 0,完成事件只触发一次;
5. 快照缓冲在 1 万次 push 中大小永不超过 `maxSnapshots`。

真正的耗时对比交给 `pnpm bench` 的 Δ% 列,由人来判断。
